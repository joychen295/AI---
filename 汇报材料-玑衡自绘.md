## 一、产品定位与概述

玑衡自绘是集成于天玑/玉衡建模工具的**AI辅助建模能力**，实现自然语言到SysML模型的智能转换，致力于降低MBSE建模门槛、提升建模效率。

### 1.1 产品背景

在MBSE（模型驱动系统工程）领域，建模是核心环节，但传统建模工具面临严峻挑战：

| 挑战类型 | 具体表现 |
|----------|----------|
| **学习成本高** | 工程师掌握专业MBSE工具平均需要6-12个月 |
| **建模效率低** | 单个中等复杂度系统建模耗时2-4周 |
| **操作繁琐** | 图形化界面操作步骤多，用户体验差 |
| **人才稀缺** | MBSE专业人才供不应求，培养周期长 |

据Gartner预测，到2027年，**70%的工程软件将内置AI能力**，AI辅助建模将成为标配功能。玑衡自绘正是顺应这一趋势，将AI能力深度融入建模工具。

### 1.2 目标用户

| 用户角色 | 使用场景 |
|----------|----------|
| **建模工程师** | 通过自然语言快速创建、修改模型元素 |
| **系统架构师** | 快速将设计思路转化为模型结构 |
| **新手用户** | 降低建模学习曲线，快速上手 |

## 二、核心功能模块

### 2.1 自然语言建模（当前版本）

#### 功能要点

**文本到模型生成**：用户输入自然语言描述，系统自动生成符合SysML规范的模型结构。

**支持的操作类型**：

| 操作类型 | 示例 | 说明 |
|----------|------|------|
| **创建** | "创建一个名为MotorController的Block，包含两个端口：输入端口powerIn和输出端口motorOut" | 根据描述自动生成模型元素 |
| **修改** | "将MotorController的powerIn端口改为双向端口" | 自动定位并修改指定属性 |
| **删除** | "删除MotorController中的motorOut端口" | 自动删除指定元素 |
| **查询** | "列出所有继承自Controller的Block" | 返回符合条件的模型元素 |

**XMI格式导出**：生成的模型可导出为XMI标准格式，无缝导入天玑/玉衡建模工具。

#### 技术实现

**基于LangGraph的工作流**：研究生团队已完成的"文本到模型"生成工作流，能够将自然语言描述转化为SysML模型，并输出XMI格式文件。

**处理流程**：

```
用户输入自然语言描述
        ↓
   语义理解与意图识别
        ↓
   模型元素生成/修改
        ↓
   XMI格式转换与校验
        ↓
   导入天玑/玉衡工具
```

#### 当前版本局限性说明

当前版本主要用于**技术能力展示**，存在以下局限：

| 局限性 | 说明 |
|--------|------|
| **黑盒化** | 用户输入描述后系统自动生成完整模型，中间过程不可见、不可干预 |
| **不可编辑** | 生成的模型无法在工作流中进行修改，必须导入工具后才能编辑 |
| **一次性生成** | 不支持增量修改，每次需要完整重新生成 |

> **说明**：上述局限将在26年6月的MBSE Copilot MVP版本中解决，届时将支持人机协同的交互式建模。

#### 解决的行业问题

| 行业痛点 | 解决方案 | 价值体现 |
|----------|----------|----------|
| SysML建模学习曲线陡峭，专业人员培养周期长 | 自然语言交互降低建模门槛 | 人员培养成本降低 |
| 手工建模效率低，中等复杂度系统建模需2-4周 | AI辅助建模，效率提升10倍 | 项目周期缩短 |
| 建模工具操作繁琐，用户体验差 | 对话式交互，简化操作流程 | 用户满意度提升 |
| MBSE专业人才稀缺 | 降低专业门槛，普通工程师也能建模 | 人才供给扩大 |

### 2.2 MBSE Copilot（规划中，26年6月MVP）

#### 功能定位

MBSE Copilot定位类似GitHub Copilot，为建模工程师提供**实时AI辅助**，将AI能力深度融入建模全过程。

#### 核心能力规划

**能力一：知识问答**

用户可随时询问MBSE相关问题，系统基于四层知识库提供专业解答：

| 知识层 | 内容 | 示例问题 |
|--------|------|----------|
| **本体论知识** | SysML规范、MBSE理论基础 | "SysML中Block和Class有什么区别？" |
| **方法论知识** | 公司内部建模方法论、最佳实践 | "如何建立状态机？" |
| **案例库** | 历史项目典型模型 | "有没有类似的控制系统参考模型？" |
| **外部知识库** | OMG官方文档、学术论文 | "SysML v2.0有哪些主要变化？" |

**能力二：代码/模型补全**

在SysML v2.0文本化建模过程中，系统自动提示补全内容：

- 用户输入"block Motor"，系统自动建议补全属性、端口定义
- 根据上下文预测下一步可能的建模操作
- 大幅提升建模效率

**能力三：智能建议**

系统分析当前模型结构，主动提供优化建议：

| 建议类型 | 说明 |
|----------|------|
| **遗漏提示** | 提示可能遗漏的必要元素 |
| **关系优化** | 发现不合理的关联关系并建议修正 |
| **规范符合性** | 检测不符合规范的建模方式 |
| **最佳实践** | 推荐更好的建模模式 |

#### 预期效果

| 能力维度 | 传统方式 | 使用MBSE Copilot后 | 效率提升 |
|----------|----------|---------------------|----------|
| 建模周期 | 2-4周 | 2-3天 | **10倍** |
| 单项目人员投入 | 3-5名建模工程师 | 1-2名 | 节约2-3人 |
| 项目交付周期 | 6个月 | 3-4个月 | **50%** |
| 人力成本 | 60-90万/项目 | 减少60%以上 | **显著降低** |

## 三、技术架构与依赖

### 3.1 技术架构

玑衡自绘基于辰极智脑AI中台构建，复用中台的核心AI能力：

```
┌─────────────────────────────────────────┐
│       天玑/玉衡建模工具 (SysML v1.6)      │
│  ┌─────────────────────────────────┐   │
│  │      玑衡自绘AI建模界面          │   │
│  └─────────────────────────────────┘   │
└────────────────┬────────────────────────┘
                 │ XMI格式交换
                 ▼
┌─────────────────────────────────────────┐
│           辰极智脑AI中台                │
│  ┌───────────┐  ┌───────────┐          │
│  │ LangGraph │  │  知识库    │          │
│  │ 工作流引擎 │  │  检索服务  │          │
│  └───────────┘  └───────────┘          │
└─────────────────────────────────────────┘
```

### 3.2 关键技术挑战

**挑战一：SysML v1.6的XMI格式对AI不友好**

| 问题 | 解决方案 |
|------|----------|
| XMI文件冗余度高、嵌套层次深 | XMI简化与预处理工具 |
| Token消耗大 | 中间格式转换（JSON/YAML） |
| 生成错误率高 | 多轮校验与自动修复 |

**挑战二：面向SysML v2.0的演进**

| SysML v1.6 | SysML v2.0 |
|------------|------------|
| XMI格式存储（XML结构） | KerML文本化建模语言 |
| 对AI不友好 | **天然适合AI处理** |
| 图形化为主 | 支持文本化建模 |
| 版本控制困难 | 可像代码一样进行Git管理 |

> **规划说明**：26年6月MVP版本将基于SysML v2.0开发，充分发挥文本化建模的优势。

### 3.3 核心设计理念：Human-in-the-Loop

玑衡自绘采用"**AI辅助、人工确认**"的设计理念：

| 设计原则 | 说明 |
|----------|------|
| **AI生成 + 人工审核** | AI生成模型草稿后，工程师审核、修改、确认后才写入正式模型库 |
| **部分接受** | 工程师可选择接受部分生成结果，拒绝或修改其他部分 |
| **可解释性** | AI建议附带推理依据，如"根据需求文档第3.2节，该接口是必需的" |
| **渐进式自动化** | 根据项目风险等级调整AI自主权限 |
